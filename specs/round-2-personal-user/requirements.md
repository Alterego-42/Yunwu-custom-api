# Requirements Document

## 1. 文档目标

本文档用于沉淀“方案A”的第二轮需求与验收标准，仅覆盖 **Phase 1：需求定义与验收标准**，不包含技术设计与任务拆分；本文已按当前 PM 已确认决策完成更新，可直接作为 Phase 2 输入。

## 2. 需求背景

### 2.1 当前仓库起点

当前仓库已经具备以下能力基础：

- 现有前端入口：`/login`、`/`、`/admin`
- 现有后端接口：登录、退出、当前用户、会话、任务、任务事件、素材上传
- 现有数据模型：用户、会话、消息、任务、素材、任务事件
- 现有实时能力：会话级 SSE 事件流
- 现有失败恢复基础：失败任务重试接口

当前仓库尚未具备以下个人用户前台能力：

- 注册页
- 首页
- 独立创建页
- 独立历史页
- 独立作品库
- 面向个人用户的失败恢复与继续创作入口

### 2.2 方案A定义

方案A是“**基于现有会话工作台能力的前台增量重组方案**”，目标是通过新增页面分层与统一语义，让个人用户可以完成：

`开始创作 -> 查看结果 -> 继续创作 -> 管理作品 -> 回看历史`

## 3. 角色定义

- 访客：未登录用户
- 个人用户：已登录的普通前台用户
- 管理员：仍保留后台能力的内部角色，不属于本轮前台主叙事

## 4. 页面职责边界

| 页面 | 主要职责 | 边界说明 |
| --- | --- | --- |
| 首页 | 进入产品后的总入口 | 只负责开始与回到最近内容，不承担复杂配置 |
| 创建页 | 发起新的生成/编辑请求 | 只负责准备输入与提交，不承担全历史浏览 |
| 工作台 | 单会话创作推进与结果承载 | 只负责当前会话，不承担全站资产管理 |
| 历史页 | 回看会话与任务过程 | 只负责过程，不承担作品主陈列 |
| 作品库 | 管理成功作品资产 | 只负责结果资产，不展示全量失败/进行中任务 |
| 登录页 / 注册页 | 处理个人用户进入与身份建立 | 不展示后台能力与创作资产 |

## 5. DESIGN SPECIFICATION

### Purpose Statement

本轮前台页面的设计目标，是把现有偏系统化的工作台重塑为面向个人创作者的持续创作空间。页面需要同时传达“可开始”“可继续”“可找回”，让用户既能快速起笔，也能在失败或中断后重新接续创作。

### Aesthetic Direction

`Editorial / magazine`

### Color Palette

- `#F4EBDD`：温暖纸张底色
- `#1E3128`：深松针绿，用于主标题与主行动
- `#B85C38`：陶土橙，用于强调与告警恢复动作
- `#C8B08A`：沙金色，用于分隔与辅助信息
- `#121212`：墨黑色，用于正文与高对比文本

### Typography

- 标题字体：`Source Han Serif SC`
- 正文字体：`IBM Plex Sans`
- 数据/状态字体：`JetBrains Mono`

### Layout Strategy

采用非对称的编辑式版面：首页与创建页使用“左侧叙事 / 右侧最近内容”的错位布局，工作台使用“主创作区 + 次信息区”的双层纵深结构，历史页与作品库使用可快速扫读的分层卡片流，避免所有页面都退化成同一种后台表格视图。

## 6. 关键产品语义

### 6.1 失败恢复

- 重试创建新任务，不覆盖原失败任务
- 系统类且可重试失败，优先提供一键重试
- 内容类失败，优先提供参数回填后再提交
- 原失败任务与后续恢复动作之间必须可追溯

### 6.2 继续创作

- 再编辑：以已有结果图作为输入素材继续编辑
- 生成变体：基于已有方向再生成新版本
- 在工作台内继续创作默认留在原会话
- 所有继续创作入口都应提供显式 fork 按钮

### 6.3 历史追溯

- 作品必须能追溯到来源任务
- 任务必须能追溯到来源提示词 / 编辑指令 / 引用素材
- 重试链与 fork 链必须能追溯前后关系

## 7. Requirements

### Requirement 1 - 登录注册与身份边界

**User Story:** 作为个人用户，我希望通过独立的登录/注册入口进入前台创作系统，而不是依赖后台或 demo 入口，这样我能拥有明确的个人身份与内容空间。

#### Acceptance Criteria

1. 当访客访问 `/login` 或 `/register` 时，云雾个人创作台应仅展示个人用户身份入口、品牌说明与必要帮助信息，不展示管理员运维信息。
2. 当未登录访客访问首页、创建页、工作台、历史页或作品库时，云雾个人创作台应重定向至登录页，并在登录成功后返回原目标页。
3. 当用户提交有效的邮箱与密码注册信息时，云雾个人创作台应创建个人账号，并以已登录态进入首页。
4. 当用户提交错误、重复或不满足规则的注册/登录信息时，云雾个人创作台应在当前页显示明确原因，并保留用户已输入的非敏感内容。
5. 在管理员或内建 demo 账号仍保留的期间，当系统识别到该类身份时，云雾个人创作台应将其前台入口与 `/admin` 后台能力保持导航分离，且普通用户默认不感知 demo 入口。

### Requirement 2 - 首页承接开始与回流

**User Story:** 作为已登录用户，我希望首页能帮我快速开始新创作，或回到最近的作品与任务，这样我不需要先理解复杂工作台结构。

#### Acceptance Criteria

1. 当用户进入首页时，云雾个人创作台应优先展示“开始创作”主入口与最近内容回流入口。
2. 当用户存在最近作品时，云雾个人创作台应在首页展示最近作品摘要，并提供进入作品库或来源工作台的入口。
3. 当用户存在最近任务或最近会话时，云雾个人创作台应在首页展示最近任务/会话摘要，并允许用户继续原会话。
4. 当用户存在失败任务时，云雾个人创作台应在首页展示失败恢复提示，并根据失败类型提供一键重试或调整后继续入口。
5. 当用户尚无任何创作记录时，云雾个人创作台应展示清晰的空状态，并把主要行动聚焦到首次创作。

### Requirement 3 - 创建页负责发起新创作

**User Story:** 作为用户，我希望在一个独立页面准备并提交新的生成或编辑请求，这样我能更专注地开始一次创作。

#### Acceptance Criteria

1. 当用户进入创建页时，云雾个人创作台应提供文生图与上传图编辑两类发起方式。
2. 当用户选择文生图时，云雾个人创作台应提供最小必要参数集，至少包含模型、尺寸或比例、数量与 Prompt。
3. 当用户选择上传图编辑时，云雾个人创作台应允许上传参考图并填写编辑指令后提交任务。
4. 当用户从首页、工作台、历史页或作品库带着来源内容进入创建页时，云雾个人创作台应自动预填来源任务的关键信息。
5. 当用户首次在创建页上传素材时，云雾个人创作台不应仅为上传动作创建空会话。
6. 当用户提交创建请求成功时，若当前尚无会话，云雾个人创作台应在提交阶段懒创建会话并将用户带入对应工作台，以便继续观察任务状态与结果。

### Requirement 4 - 工作台负责单会话创作推进

**User Story:** 作为用户，我希望工作台聚焦在一个会话内的任务进展、结果和下一步操作，这样我能连贯地推进一次创作旅程。

#### Acceptance Criteria

1. 当用户进入某个工作台时，云雾个人创作台应展示该会话下的消息、任务、事件与相关素材。
2. 当会话内存在进行中任务时，云雾个人创作台应持续展示任务状态变化，并在实时能力不可用时退化为轮询而不中断当前视图。
3. 当任务成功完成时，云雾个人创作台应在工作台中直接展示结果，并提供再编辑、生成变体、下载、查看来源信息与 fork 入口。
4. 当任务失败时，云雾个人创作台应在工作台中展示失败原因、失败状态与后续操作入口，而不是只停留在技术状态词。
5. 当用户在工作台内继续创作时，云雾个人创作台应默认在原会话内创建后续任务，以保持上下文连续。
6. 当用户明确选择 fork 时，云雾个人创作台应以当前来源任务为起点创建新会话，而不是改写原会话链路。

### Requirement 5 - 历史页负责过程回看

**User Story:** 作为用户，我希望在历史页回看自己做过的会话与任务过程，这样我能快速找到失败点、继续中的任务或过去的思路。

#### Acceptance Criteria

1. 当用户进入历史页时，云雾个人创作台应按时间顺序展示会话或任务历史，并区分成功、失败与进行中状态。
2. 当用户查看某条历史记录时，云雾个人创作台应展示足以识别该记录的摘要信息，包括时间、状态、任务类型与来源描述。
3. 当用户从历史页选择某条记录继续处理时，云雾个人创作台应提供返回来源工作台的入口。
4. 当用户需要处理失败任务时，云雾个人创作台应在历史页根据失败类型提供一键重试或参数回填后的继续路径。
5. 当用户没有可展示的历史内容时，云雾个人创作台应展示空状态，并引导用户去创建页开始创作。

### Requirement 6 - 作品库负责成功作品管理

**User Story:** 作为用户，我希望在作品库集中管理成功作品，而不是在历史流里翻找所有任务，这样我能更高效地复用和下载成果。

#### Acceptance Criteria

1. 当用户进入作品库时，云雾个人创作台应默认仅展示成功产出的作品。
2. 当作品出现在作品库中时，云雾个人创作台应为每个作品提供预览、下载、删除、查看来源任务、继续创作与 fork 入口。
3. 当用户查看作品来源时，云雾个人创作台应能够把用户带到来源任务或来源工作台，而不是停留在孤立图片预览。
4. 当作品被删除时，云雾个人创作台应将其从作品库默认列表移除，但保留必要的历史追溯关系。
5. 当用户没有任何成功作品时，云雾个人创作台应展示作品库空状态，并引导用户去创建页或首页开始创作。

### Requirement 7 - 失败恢复

**User Story:** 作为用户，我希望在任务失败后能明确知道为什么失败、能否重试、下一步去哪，这样失败不会打断我的创作节奏。

#### Acceptance Criteria

1. 当任务失败时，云雾个人创作台应展示可被用户理解的失败原因，而不是仅展示底层状态码或空白失败状态。
2. 当失败被识别为系统类且允许重试时，云雾个人创作台应提供“一键重试”入口，并在触发后创建新的任务记录而不是覆盖原任务。
3. 当系统创建了重试任务时，云雾个人创作台应保留“原失败任务 -> 新重试任务”的可追溯关系。
4. 当失败被识别为内容类时，云雾个人创作台应提供“带原参数回到创建页调整后再提交”的恢复路径。
5. 当失败属于系统配置异常且当前不允许用户直接恢复时，云雾个人创作台应展示明确说明，并避免把不可用动作伪装成可点击成功路径。
6. 当用户从首页、历史页或工作台发起失败恢复时，云雾个人创作台应保持恢复入口语义一致，不让用户重新理解不同页面规则。

### Requirement 8 - 继续创作

**User Story:** 作为用户，我希望在拿到结果后立刻继续做变体或编辑，这样我能把一次灵感逐步推到更满意的版本。

#### Acceptance Criteria

1. 当任务成功产生结果图时，云雾个人创作台应提供“再编辑”“生成变体”和“fork 新会话”三类后续入口。
2. 当用户选择“再编辑”时，云雾个人创作台应把被选中的结果图作为后续编辑任务的输入素材。
3. 当用户选择“生成变体”时，云雾个人创作台应沿用来源任务的核心方向，并允许用户在继续提交前补充或调整输入。
4. 当用户从工作台中发起继续创作时，云雾个人创作台应默认沿用原会话上下文。
5. 当用户从历史页或作品库发起继续创作时，云雾个人创作台应将用户带回来源工作台或带预填信息进入创建页，而不是丢失来源上下文。
6. 当用户显式选择 fork 时，云雾个人创作台应以来源任务为分支起点创建新会话，并保留与来源会话的最小必要关联。

### Requirement 9 - 历史追溯与链路完整性

**User Story:** 作为用户，我希望任何作品和任务都能追溯来源，这样我在复盘或复用时不需要凭记忆找线索。

#### Acceptance Criteria

1. 当用户查看任一作品时，云雾个人创作台应允许其追溯到来源任务。
2. 当用户查看任一任务时，云雾个人创作台应允许其查看来源提示词、编辑指令与引用素材摘要。
3. 当任务属于重试链的一部分时，云雾个人创作台应允许用户识别它是“由哪次失败重试而来”或“后来重试到了哪里”。
4. 当任务或会话属于 fork 链的一部分时，云雾个人创作台应允许用户识别它来自哪条来源会话 / 来源任务。
5. 当用户从历史页、作品库或工作台跨页面跳转时，云雾个人创作台应保持来源关系不丢失。
6. 当作品被从作品库删除时，云雾个人创作台应仍然保留最小必要的历史链路，以避免会话追溯断裂。

### Requirement 10 - 前后台分离

**User Story:** 作为 PM 和用户，我希望个人用户前台与内部后台保持清晰分离，这样产品叙事不会再次滑回后台系统视角。

#### Acceptance Criteria

1. 当普通个人用户浏览前台页面时，云雾个人创作台应不展示后台管理、provider 检查、告警确认等内部运维能力。
2. 当管理员需要处理模型、任务诊断或 provider 问题时，云雾个人创作台应将相关能力保留在 `/admin` 路径下。
3. 当同一系统同时存在前台与后台时，云雾个人创作台应在导航、命名与跳转关系上清楚区分两者。
4. 当第二轮页面被评审时，云雾个人创作台应以个人创作体验为主叙事，而不是以内控诊断逻辑组织前台信息架构。

## 8. 已确认决策与当前边界

### 8.1 已确认决策

| 决策项 | 已确认值 | 影响范围 |
| --- | --- | --- |
| 注册方式 | 自注册 | 登录 / 注册页、认证后端、用户模型 |
| 创建页何时创建会话 | 首次提交任务时懒创建 | 创建页、上传链路、任务创建接口 |
| 继续创作默认归属 | 默认归原会话 | 工作台、历史页、作品库 |
| 是否提供 fork | 提供显式 fork 按钮 | 工作台、历史页、作品库、追溯链 |
| 作品删除语义 | 作品库软删除，历史链保留 | 作品库、历史追溯 |
| 失败恢复策略 | 系统类失败一键重试；内容类失败参数回填 | 首页、工作台、历史页、创建页 |

### 8.2 当前边界假设

- P0 不做邮箱验证
- P0 不做忘记密码
- demo 账号仅内部保留，不作为普通用户正式入口
