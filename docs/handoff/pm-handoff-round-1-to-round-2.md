# PM 交接包：第一轮收口到第二轮启动

## 1. 交接目的

本文档用于新 PM 接手项目时快速建立完整上下文，包括：

- 项目现在是什么
- 第一轮做到了什么程度
- 哪些事情已经被验证
- 哪些方向已经明确不做
- 第二轮为什么要转向个人用户前台产品

## 2. 项目一句话概述

这是一个以 NestJS + Vite React 为基础的图像生成/编辑平台项目。第一轮完成了后端能力、任务链路、管理后台、Docker 交付和回归资产；第二轮将从“可跑的系统”升级为“面向个人用户的 AI 图片生成 + 编辑工作台”。

## 3. 第一轮已完成内容

### 3.1 工程与架构

- 后端：NestJS
- 前端：Vite + React + Tailwind + shadcn/ui 风格组件
- ORM：Prisma
- 数据库：PostgreSQL
- 缓存 / 队列：Redis + BullMQ
- 对象存储：MinIO
- 实时能力：SSE
- 运行方式：本地开发 + Docker 单机路径

### 3.2 第一轮产品/系统能力

- 单 provider 图像生成/编辑链路
- 认证能力（管理员 + demo）
- 会话、任务、事件流
- Worker 消费与异步任务执行
- 对象存储与图片结果管理
- provider 健康检查 / 测试 / 模型快照
- provider 告警闭环与单条确认
- 管理后台基础运维能力
- 本地回归脚本
- Docker 运维交接文档
- 图片化最终回归证据

### 3.3 第一轮交付性质

第一轮更偏“系统落地 + 工程打底 + 运维闭环验证”，不是完整的个人用户产品。它已经证明平台能力真实可跑，但个人用户前台体验仍未成型。

## 4. 第一轮关键验证结论

### 4.1 功能与链路

- 本地链路验证通过
- 容器内业务闭环验证通过
- provider 管理面、告警闭环、任务链路已跑通

### 4.2 部署与运维

- Docker 单机路径可运行
- Docker Hub 拉取依赖宿主机代理 `127.0.0.1:7897`
- Prisma / OpenSSL 告警已通过基础镜像调整处理

### 4.3 交付资产

- 首轮 freeze 文档已整理
- QA 回归脚本已沉淀
- 图片化回归证据已生成
- Git 远端仓库已完成首推到 `main`

## 5. 已确认的关键决策

### 5.1 技术方向

- 后端继续使用 NestJS
- 前端继续使用 Vite + React
- Prisma、PostgreSQL、Redis、MinIO、BullMQ 保持不变
- Docker 单机交付路径继续保留

### 5.2 产品约束

- 当前只支持单 provider
- 当前不做多租户
- 当前不做复杂计费
- 当前不做社区 / 社交 / 开放平台

### 5.3 产品定位纠偏

这个项目最初在推进过程中一度偏向“后台/运维系统”视角。后续访谈已明确纠偏：**第二轮应以个人用户为中心，而不是以内部管理员为中心。**

## 6. 第二轮方向总结

### 6.1 第二轮一句话方向

面向个人用户的 AI 图片生成 + 编辑工作台。

### 6.2 第二轮核心闭环

`生成 -> 编辑 -> 保存 -> 继续创作`

### 6.3 第二轮主场景

- 文生图
- 上传图后编辑
- 历史作品回看与复用

### 6.4 第一批优先包

- 注册 / 登录
- 新建生成
- 上传图编辑

### 6.5 第二轮非目标

- 多租户
- 多 provider 正式能力
- 复杂计费系统
- 社交 / 社区
- 开放平台 / API 商业化
- 第二轮移动端专项版本

## 8. 新 PM 最该知道的约束

- 第一轮留下的是一个“有后台支撑能力的系统底座”，不是完整的 C 端产品
- 第二轮需要把产品叙事从后台运维切换到个人用户创作体验
- 已有管理后台不要推倒重来，应作为支撑系统保留
- 当前 provider、部署方式、技术栈都已经相对稳定，不适合第二轮再做大规模基础设施迁移

## 9. 风险与信息缺口

### 9.1 风险

- 前后台分离涉及路由、权限、数据隔离
- 账号体系引入后，原有管理员 / demo 逻辑需要重新梳理边界
- 结果页、作品库、历史页如果定义不清，页面职责会重复
- 如果第二轮继续优先后台能力，会偏离个人用户产品方向

### 9.2 信息缺口

当前已经确认了第二轮大方向和核心流程，但以下内容仍可在后续版本细化：

- 模板 / 预设是否进入后续小版本
- 作品库是否扩展收藏、标签、搜索
- 失败任务是否扩展为参数回填
- 移动端专项优化的节奏

## 10. 推荐接手顺序

新 PM 建议按以下顺序接手：

1. 先理解第一轮工程资产和系统边界
2. 再接受“第二轮面向个人用户”的定位纠偏
3. 阅读第二轮 PRD v0.1
4. 以用户旅程而不是技术模块拆解第二轮任务
5. 先做注册登录、新建生成、上传图编辑三个优先包

## 11. 工具链注意事项

### 11.1 已知注意点

- PowerShell 存在 `ConvertFrom-Json -Depth` 兼容性问题，后续脚本和自动化尽量使用 Node / Python 解析 JSON
- Docker 拉取镜像依赖宿主机代理 `127.0.0.1:7897`
- 截图链路未直接依赖 `require('playwright')`，当前走 Chrome Headless + CDP
- 其他工具链问题可继续参考 `docs/qa/toolchain-issues.md`

### 11.2 对新 PM 的建议

不要把工具链问题误判为产品问题，也不要因为现有后台能力完整，就把第二轮继续做成“给管理员用”的系统。

## 12. 结论

第一轮已经完成了“系统底座 + 运维闭环 + 验证资产”。第二轮的关键不是再扩后台，而是把系统能力重新组织成面向个人用户的创作产品。新 PM 的核心任务，是守住这个定位，不让项目重新滑回后台系统思路。
