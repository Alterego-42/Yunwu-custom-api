## 2. 方案A一句话结论

方案A不推翻第一轮已验证的“会话 + 任务 + 事件流 + 后台支撑”底座，而是在其之上补齐**个人用户前台页面分层、创作闭环语义和内容追溯规则**，把现有能力从“能跑的系统”重组为“能持续使用的个人创作产品”。

## 3. 当前仓库现状（用于锁定第二轮起点）

基于当前仓库代码与文档，第二轮需求起点已确认如下：

### 3.1 已具备能力

- 前端已存在 `/login`、`/`、`/admin` 三类入口，其中 `/` 当前承载工作台能力，`/admin` 承载内部管理后台。
- 后端已具备：
  - 登录、会话态恢复、退出登录
  - 会话列表 / 会话详情
  - 文生图 / 图编辑任务创建
  - 上传素材
  - 任务详情
  - 会话级 SSE 事件流
  - 任务事件历史
- 数据模型已具备 `User / Conversation / Message / Task / Asset / TaskEvent` 关系，说明“历史追溯”和“继续创作”已有底层承载基础。
- 失败任务已存在 `POST /api/tasks/:id/retry` 能力，且当前语义是：**失败任务不会被覆盖，而是创建一条新的重试任务，并保留前后关联事件**。
- `Asset.status` 已包含 `deleted` 枚举，说明作品库“软删除”可以优先复用现有状态模型，而不是另起一套作品表。

### 3.2 当前缺口

- 前台仍缺少独立的：
  - 首页
  - 创建页
  - 历史页
  - 作品库
  - 注册页
- 当前登录体系仍是**内建 `admin/demo` 账号模型**，尚未形成面向个人用户的正式自注册入口。
- 当前失败重试入口主要服务后台诊断场景，尚未变成面向个人用户的可理解操作。
- 当前工作台更像“会话调试台”，还不是职责清晰的“创作工作台 / 结果页”。

### 3.3 第二轮产品约束

- 不推翻后台，不把 `/admin` 并入前台。
- 不改多 provider / 多租户 / 复杂计费方向。
- 不把第二轮做成“后台前移”的换皮版本。

## 4. 方案A产品判断

### 4.1 为什么选方案A

方案A选择**增量重组**而不是**重建一套全新前台体系**，原因有三：

1. 第一轮已经验证了任务链路、事件流、对象存储和后台诊断闭环，具备可复用价值。
2. 当前数据结构天然支持“会话—任务—作品—事件”的创作追溯，不需要为第二轮重造核心模型。
3. 第二轮最大风险不在基础设施，而在**页面职责混乱、前台体验系统感过重、失败与继续创作语义不清**。

### 4.2 方案A的核心原则

- **前台分层**：把“开始创作”“继续创作”“回看历史”“管理作品”拆到不同页面。
- **会话主轴**：创作仍以会话为承载单位，避免结果与上下文脱钩。
- **结果可回流**：任一成功作品都应能回到创作链路继续生成、再编辑或追溯来源。
- **失败可恢复**：失败不是死路，必须有明确原因、明确去向、明确下一步。
- **后台保留**：后台继续负责诊断、模型管理、告警与运维，不与个人用户前台混用。

## 5. 第二轮目标与成功标准

### 5.1 目标

- 建立面向个人用户的创作闭环：`开始 -> 生成/编辑 -> 查看结果 -> 继续创作 -> 留存 -> 回看`
- 让用户理解“作品、历史、工作台”三者关系，而不是在单页里堆功能
- 将失败恢复从“工程诊断动作”转成“用户可理解动作”
- 让前台从后台系统感切换为创作产品感

### 5.2 成功标准

- 新用户在 5 分钟内可以完成一次注册（或登录）、一次生成、一次继续创作。
- 用户能区分首页、创建页、工作台、历史页、作品库的职责，不需要反复试错。
- 用户能从任一作品追溯到来源任务、提示词/编辑指令与所属会话。
- 用户在任务失败后，能理解失败原因，并找到“重试”或“调整后再创作”的入口。

## 7. 信息架构（方案A）

- `/`：首页
- `/login`：登录
- `/register`：注册
- `/create`：创建页
- `/workspace/:conversationId`：工作台
- `/history`：历史页
- `/library`：作品库
- `/admin`：后台管理页

> 关键原则：  
> 首页负责“开始/回到最近”；创建页负责“发起”；工作台负责“执行中与继续创作”；历史页负责“过程回看”；作品库负责“结果管理”。

## 8. 页面职责边界

| 页面 | 核心职责 | 应该承载 | 不应承载 |
| --- | --- | --- | --- |
| 首页 | 进入产品后的总入口 | 新建 CTA、最近作品、最近任务、最近会话、状态提示 | 完整参数编辑、完整历史时间流、复杂作品管理 |
| 创建页 | 发起一次新的创作请求 | 文生图参数、上传图编辑入口、来源内容预填、提交前确认 | 全量历史浏览、长时间任务观察、作品沉淀管理 |
| 工作台 | 承载单个会话中的创作推进 | 任务状态、事件流、结果查看、继续创作、失败恢复 | 站内总览、跨会话检索、作品批量管理 |
| 历史页 | 回看用户做过什么 | 会话流、任务流、状态筛选、失败/成功识别、回到工作台 | 作品主陈列、下载/删除的批量管理 |
| 作品库 | 管理用户已产出的作品 | 成功作品卡片、来源追溯、下载、删除、回到来源任务 | 所有失败/进行中任务、完整事件日志 |
| 登录/注册页 | 完成个人用户身份进入 | 登录、注册、身份说明、跳转恢复 | 创作内容浏览、后台能力入口、任务列表 |

## 9. 核心对象定义

### 9.1 会话

- 会话是用户一次连续创作旅程的容器。
- 一个会话下可以包含多次任务，包括首次生成、编辑、变体、重试。
- 工作台以会话为主视角展开。

### 9.2 任务

- 任务是一次具体的生成或编辑执行。
- 任务有状态流转、事件流、输入摘要、失败原因、结果素材。
- 历史页关注任务过程，作品库只关注成功产出。

### 9.3 作品

- 作品是成功产出的结果图片。
- 作品必须可回看来源任务与来源输入。
- 作品是用户沉淀资产，不等于完整历史。

### 9.4 历史

- 历史是会话、任务、任务事件的可回看集合。
- 历史不等于作品；失败和进行中也属于历史，但不默认进入作品库。

## 10. 三个关键语义

### 10.1 失败重试语义

方案A明确：

- “重试”不是覆盖原任务，而是**在原会话中创建一条新的任务记录**。
- 原失败任务必须保留，且用户可看见“失败任务 -> 重试任务”的关联关系。
- 重试默认继承上一次任务的输入参数、提示词、素材引用与能力类型。
- **系统类且可重试失败**优先给“一键重试”。
- **内容类失败**优先给“带原参数回到创建页调整后再提交”。
- 若为明确不可重试的系统配置异常，则展示原因与禁用态提示，不伪装成可恢复操作。

**产品解释语：**

- 一键重试：同样的输入，再试一次。
- 调整后继续：带着原参数回到创建页，改完再提交。

### 10.2 继续创作语义

方案A明确把“继续创作”拆成三类入口：

1. **再编辑**
   - 以某张已有结果图为输入素材，再发起编辑任务。
   - 适合“改局部风格 / 改文案元素 / 改背景 / 改构图方向”等场景。

2. **生成变体**
   - 以某张已有结果图或其来源提示词为基础，发起新的生成任务。
   - 适合“保留方向但再给几版”场景。

3. **Fork 新会话**
   - 以当前方向为起点，复制来源上下文，但在新会话中继续探索。
   - 适合“我想保留原会话完整脉络，同时开一条平行分支”的场景。

默认规则：

- 在工作台内发起继续创作时，默认留在**原会话**中，保持上下文连续。
- 在历史页 / 作品库发起继续创作时，默认跳回**来源工作台**，并带入来源任务信息。
- 所有继续创作入口都应同时提供显式 **fork 按钮**，允许用户主动切换到新会话分支。

### 10.3 历史追溯语义

方案A要求至少支持以下追溯路径：

- 作品 -> 来源任务
- 来源任务 -> 来源提示词 / 编辑指令 / 引用素材
- 来源任务 -> 所属会话
- 失败任务 -> 后续重试任务
- Fork 会话 -> 来源任务 / 来源会话

换言之，用户不能只看到一张孤立图片；每张作品都应能回答四个问题：

1. 这张图从哪来？
2. 当时用了什么输入？
3. 后来是否继续创作过？
4. 这次继续创作是在原会话还是 Fork 出去的？

## 11. 页面详细要求补充

### 11.1 首页

- 首页是“个人创作入口页”，不是“迷你工作台”。
- 首页应优先给出两个方向：
  - 去开始新的创作
  - 去继续最近一次创作
- 首页展示内容建议优先级：
  1. 新建创作入口
  2. 最近作品
  3. 最近任务 / 最近会话
  4. 若存在失败任务，则给恢复提醒

### 11.2 创建页

- 创建页是“起一笔”的页面。
- 创建页需要同时覆盖：
  - 文生图
  - 上传图后编辑
- 创建页应支持来源预填：
  - 从首页点击“继续上次方向”
  - 从工作台点击“生成变体”
  - 从历史页 / 作品库点击“调整后继续”
- 创建页采用**懒创建会话**：首次上传仅保存素材，首次提交任务时才真正创建会话。

### 11.3 工作台

- 工作台是“单会话创作进行页 + 结果页”。
- 工作台必须承载：
  - 当前任务状态
  - 历史消息/事件
  - 成功结果查看
  - 失败原因与恢复入口
  - 继续创作操作
  - 显式 Fork 按钮
- 工作台不承担全站总览职能。

### 11.4 历史页

- 历史页强调“过程”，不是“结果精选”。
- 历史页需要让用户快速识别：
  - 哪些任务成功
  - 哪些任务失败
  - 哪些任务仍在进行中
  - 哪些会话值得回到工作台继续推进

### 11.5 作品库

- 作品库强调“结果资产”，不是“全量操作日志”。
- 作品库默认只展示成功作品。
- 每张作品至少要支持：
  - 预览
  - 查看来源任务
  - 查看来源输入
  - 下载
  - 删除
  - 继续创作
  - Fork 新会话

### 11.6 登录 / 注册页

- 登录 / 注册页必须完全站在个人用户视角表达，不复用后台叙事。
- 不应把管理员、诊断、provider 状态等概念暴露给普通用户。
- 注册页支持**个人用户自注册**，不再要求通过内建 demo 账号进入前台。

## 12. 删除、留存与可见性规则

### 12.1 作品删除

方案A明确采用**作品库软删除**：

- 从作品库删除后，作品不再出现在默认作品库列表中。
- 历史中的任务记录与追溯关系保留，避免创作链断裂。
- 删除的是“作品库展示资格”，不是直接抹掉来源任务、事件与会话关系。

### 12.2 历史保留

- 失败任务、重试任务、成功任务都属于历史。
- 作品库只负责结果资产，不替代历史页。

### 12.3 进行中任务可见性

- 进行中任务优先在工作台与首页提醒中暴露。
- 作品库默认不收录进行中任务。

## 13. 风险、缺口与依赖

### 13.1 已识别风险

- 当前前端路由仍以工作台单页为主，页面分层落地后导航与跳转关系会明显增加。
- 当前鉴权仍以内建 `admin/demo` 为主，个人用户注册接入会改变前后台身份边界。
- 当前重试能力虽已存在，但接口与入口仍偏后台语义，需要前台产品化重述。
- 如果历史页与作品库边界不稳，会再次出现“信息重复但都不够用”的问题。
- 当前失败分类已在任务事件中沉淀，但前台尚未形成“系统类失败 / 内容类失败”的统一动作映射。

### 13.2 本轮剩余边界项

- P0 是否做邮箱验证：当前仍按“不做邮箱验证”进入技术设计，不阻塞本轮闭环。
- P0 是否做忘记密码：当前仍按“不做忘记密码”进入技术设计，不阻塞本轮闭环。
- 首页内容推荐深度与视觉精修：留在后续页面打磨阶段处理。

## 14. 已确认产品决策（进入设计基线）

以下项目已由 PM 明确拍板，可直接作为 Phase 2 技术方案输入：

| 决策项 | 已确认值 | 设计含义 |
| --- | --- | --- |

| 失败恢复策略 | 系统类失败一键重试；内容类失败参数回填 | 前台 CTA 需要按失败类型分流 |

## 16. 结论

第二轮不缺底层能力，真正缺的是**前台页面分工、用户动作语义和创作资产组织方式**。方案A的核心价值，在于以最小重构成本，把第一轮的系统能力变成个人用户可理解、可持续回访、可继续创作的产品体验；当前文档中的关键待拍板项已经收敛到可进入技术设计的程度。
