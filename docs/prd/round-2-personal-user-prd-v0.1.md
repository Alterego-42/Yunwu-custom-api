## 2. 产品定位与背景

### 2.1 定位

第二轮产品定位纠偏为：**面向个人用户的 AI 图片生成 + 编辑工作台**。

第一轮已经完成了后端能力、管理后台、任务链路、Docker 交付和基础回归能力，证明系统“可跑、可测、可运维”。第二轮不再以后台能力堆叠为主，而是把前台产品体验补齐，让个人用户可以顺畅完成创作闭环。

### 2.2 背景判断

第一轮形成的能力基础包括：

- 单 provider 图像生成/编辑能力
- 任务队列、异步执行、状态追踪、事件流
- 对象存储、结果回显、回归脚本、Docker 路径
- provider 检查、测试、告警、任务诊断、管理后台

这些能力已经足以支撑前台用户产品继续建设。第二轮的重点不是再扩 provider 或多租户，而是把现有能力重组成个人用户可理解、可持续使用的创作产品。

## 3. 目标用户

### 3.1 核心用户

- 高频创作者
- 设计师
- 自媒体内容生产者
- 有持续图片创作需求的个人用户

### 3.2 用户特征

- 关注结果质量和出图效率
- 需要反复修改、迭代，而不是一次性出图
- 需要方便找回历史作品、任务和灵感上下文
- 对失败敏感，希望知道失败原因并快速重试

## 4. 第二轮目标与成功标准

## 4.1 产品目标

- 建立个人用户创作闭环：`生成 -> 编辑 -> 保存 -> 继续创作`
- 提升首次上手成功率
- 提升任务稳定性和失败恢复体验
- 建立可追溯的历史记录与作品沉淀能力

### 4.2 成功标准

- 新用户 5 分钟内完成“生成一张 + 编辑一张”
- 用户能看懂任务状态、失败原因和处理入口
- 用户能在结果页继续创作，而不是只能下载离开
- 用户能在历史页和作品库中找回自己的内容
- 系统稳定性、可恢复性显著优于第一轮

## 5. 核心场景

### 5.1 文生图

用户输入 Prompt，选择基础参数，快速得到第一张图。

### 5.2 上传图编辑

用户上传已有图片，输入编辑指令，对原图进行继续创作。

### 5.3 历史回看与复用

用户回到历史会话、任务或作品，重新查看来源、继续编辑或生成变体。

## 6. 范围定义

### 6.1 P0

- 个人账号注册/登录（邮箱 + 密码）
- 前后台分离
- 文生图主流程
- 上传图编辑主流程
- 独立历史页
- 作品库最小版
- 失败原因展示 + 一键重试
- 结果页继续创作

### 6.2 P1

- 首页内容组织优化
- 参数配置体验优化
- 任务状态与反馈优化
- 历史/作品浏览体验优化
- 创作型 UI 细节优化

### 6.3 P2

- 模板/预设
- 更复杂的编辑工具
- 移动端专项优化
- 额度/计费相关能力

## 7. 非目标

第二轮明确不做：

- 多租户
- 多 provider 正式能力
- 复杂计费系统
- 社区/社交
- 开放平台 / API 商业化
- 移动端专项版本

## 8. 信息架构

- `/`：首页
- `/login`：登录
- `/register`：注册
- `/create`：新建任务
- `/library`：作品库
- `/history`：历史记录
- `/workspace/:conversationId`：创作工作台 / 结果页
- `/admin`：管理后台

原则：

- 前台服务个人用户
- 后台保留内部管理与排障能力
- 两者路径分离，但共享底层服务能力

## 9. 页面与核心流程

### 9.1 首页

首页优先展示：

- 新建入口
- 最近作品
- 最近任务

首页目标不是承载复杂操作，而是让用户快速开始或回到最近内容。

### 9.2 新建页

最小参数集：

- 模型
- 尺寸 / 比例
- 张数
- Prompt

同时提供上传图编辑入口。高级参数不作为第二轮 P0 必做项。

### 9.3 上传图编辑页

最小链路：

- 上传图片
- 输入编辑指令
- 提交任务

本轮不做局部遮罩、多图输入等高复杂能力。

### 9.4 作品库

默认按**作品卡片流**组织。

每张作品应支持：

- 查看来源任务
- 查看来源 Prompt / 指令
- 下载
- 删除

### 9.5 历史页

历史记录独立成页，不与作品库完全混合。

历史页用于承载：

- 会话时间流
- 任务时间流
- 成功 / 失败 / 进行中状态识别

### 9.6 结果页 / 工作台

结果页是第二轮关键页面，不应只做静态展示。

必须提供继续创作入口：

- 再编辑
- 生成变体

## 10. 功能需求

### 10.1 账号体系

- 支持邮箱 + 密码注册
- 支持邮箱 + 密码登录
- 用户前台只看自己的会话、任务、作品
- 管理后台与用户前台入口分离

### 10.2 文生图

- 支持最小参数创建任务
- 创建后可查看状态变化
- 成功后结果进入结果页 / 作品库

### 10.3 上传图编辑

- 支持上传原图
- 支持输入编辑指令
- 支持提交编辑任务
- 成功后结果可继续创作

### 10.4 历史追溯

追溯层级至少包含：

- 会话
- 任务
- 作品

用户应能从作品回到来源任务，再看到来源输入。

### 10.5 作品库

最小版能力：

- 保存结果图
- 查看来源任务 / 提示词
- 下载
- 删除

### 10.6 失败恢复

- 明确展示失败原因
- 提供一键重试入口
- 支持快速回到继续创作链路

### 10.7 继续创作

结果页必须支持：

- 再编辑
- 生成变体

## 11. UI / 体验原则与优化时机

### 11.1 体验原则

- 优先可用，不先追求过度视觉包装
- 目标强度为：**可用 + 有产品感**
- 前台页面必须弱化后台系统感，强化创作产品感

### 11.2 当前策略

当前决策是不单独先出 UI 方案，而是先写功能 PRD。但本 PRD 仍约束页面结构与体验方向，避免后续返工。

### 11.3 UI 优化时机

- P0 开发前：定信息架构、页面骨架、导航关系
- P0 开发中：同步做可用级 UI
- P0 验收后：补视觉精修

## 12. 开发拆分建议

第二轮按**用户旅程**切分，而不是只按技术模块切分。

建议顺序：

1. 注册 / 登录
2. 新建生成
3. 上传图编辑
4. 作品库 / 历史页
5. 失败恢复
6. 结果页继续创作

第一批优先包已确认：

- 注册登录
- 新建生成
- 上传图编辑

## 13. 验收标准

- 注册 / 登录可用
- 文生图链路可用
- 上传图编辑链路可用
- 作品库支持查看、下载、删除
- 历史页支持回看任务与会话
- 失败任务支持一键重试
- 结果页支持再编辑和生成变体
- 前后台入口已分离

## 14. 风险与待确认项

### 14.1 已识别风险

- 前后台分离会影响路由、权限和数据归属
- 账号体系引入后，用户数据隔离要求明显提升
- 结果页、历史页、作品库若边界不清，容易重复设计
- 若前台继续沿用后台式界面，个人用户产品感会不足

### 14.2 待后续版本确认

- 模板 / 预设是否进入下一小版本
- 作品库是否引入收藏、标签、搜索
- 失败重试是否扩展为参数回填
- 是否在第三轮再考虑移动端专项优化
