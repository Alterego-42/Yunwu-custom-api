# v0.5.0 User-Side Update Design

## 1. 文档目标

本文定义 `v0.5.0` 的用户侧更新策略。当前工作树可暂定为 `v0.5.0`：它包含批量并发图片请求、批量任务数据表、普通任务/批量任务队列拆分、首页布局修正，以及桌面 Compose 默认并发策略调整。

核心结论：

- `v0.4.3 -> v0.5.0` 必须作为桌面包更新处理，不能只做镜像 tag 更新。
- `v0.5.0` 应成为第一个“更新感知”的桌面基线。
- `v0.5.0` 首版只建议实现检查更新和引导下载；镜像栈更新可以按本文接口和状态机预留，随后再打开。
- 真正替换 Electron portable zip 不在 `v0.5.0` 内自动完成，除非后续切换到 installer 或外部 updater helper。

## 2. 设计输入

已确认上一版策略：

- `v0.4.3` 发布为 Windows x64 Electron portable zip。
- API、Worker、Web 发布为 GHCR 镜像。
- 桌面壳启动时写入 runtime `.env`，并使用内置 Compose 文件执行 `docker compose pull` 和 `docker compose up -d`。
- `v0.4.3` 的 `YUNWU_IMAGE_TAG` 在桌面主进程和桌面 Compose 默认值中硬编码为 `v0.4.3`。

当前工作树新增事实：

- 新增 `TaskBatchItem` 表和 Prisma migration：`20260605190000_add_task_batch_items`。
- `batchCount > 1` 会创建一个父任务和多个批量执行槽。
- 普通任务 worker 默认并发从旧桌面路径的 `2` 提升到 `50`。
- 批量父任务有独立队列，默认并发为 `2`。
- Web 的工作台、作品库、首页、历史页和管理台都已经批量感知。
- 批量任务不能 fork；批量再编辑必须先选择一个成功图片。

这些变化意味着 `v0.5.0` 不只是业务 UI 更新，也包含运行时配置和数据库 schema 更新。

## 3. 发布兼容性判断

### 3.1 `v0.4.3 -> v0.5.0`

必须标记为 `requiresDesktopUpdate=true`。

原因：

- `v0.4.3` 桌面壳没有更新检查和镜像 tag 持久化能力。
- `v0.4.3` 每次启动都会写死 `YUNWU_IMAGE_TAG=v0.4.3`。
- 旧桌面 runtime `.env` 会继续给 `TASK_WORKER_CONCURRENCY` 写入旧值 `2`，不满足 `v0.5.0` 普通任务并发目标。
- `v0.5.0` 需要新 migration，虽然是新增表的前向迁移，但回滚不能承诺自动安全。
- 用户必须拿到新桌面壳，才能拥有后续更新能力。

用户路径：

1. 关闭旧版 `Yunwu Desktop`，确保旧实例执行 `docker compose down`。
2. 下载并解压 `Yunwu.Desktop-0.5.0-win-x64-portable.zip`。
3. 启动新版桌面壳。
4. 新版桌面壳使用 `v0.5.0` 镜像启动服务栈。
5. API 容器启动时执行 `prisma migrate deploy`，创建批量任务表。
6. 旧数据、账号、登录 secret、Docker volume 保留。

### 3.2 `v0.5.0 -> v0.5.x`

允许按 release manifest 判断：

- 桌面壳、Compose schema、runtime env、IPC 或 updater 逻辑有变化：`requiresDesktopUpdate=true`。
- 只变更 API/Worker/Web 镜像且兼容现有 Compose/runtime：可允许 `imageOnlySupported=true`。
- 有数据库 migration：允许前向更新，但必须显示风险确认；不承诺自动回滚。
- 有破坏性 migration 或需要手工步骤：`updateBlocked=true`，只引导用户查看 release notes。

## 4. 产品体验策略

### 4.1 更新入口

更新能力优先属于桌面壳，不属于业务账号系统。

入口优先级：

1. 桌面启动页：服务启动前后均可显示更新状态。
2. 桌面错误页：如果启动失败且发现新版本，提示用户下载新版。
3. Web 设置页或管理页：可选。仅通过受限 `window.yunwuDesktop` IPC 读取更新状态，不直接操作 Docker。

不依赖 `member/admin/demo` 登录态检查更新。原因是用户可能在 API 未启动、Cookie 失效或数据库迁移失败时仍需要更新提示。

### 4.2 用户可见状态

建议状态文案：

- `当前已是最新版本`
- `发现新版 v0.5.1`
- `此版本需要下载新的桌面包`
- `可直接更新本地服务镜像`
- `更新会重启本地服务并保留数据`
- `此版本需要手动处理，请查看发布说明`
- `更新检查失败，可继续使用当前版本`

更新提示必须是非阻塞的。除非 manifest 标记 `minimumSupportedDesktopVersion` 高于当前桌面版本，否则不要阻止用户进入工作台。

### 4.3 操作分级

`v0.5.0` 首版推荐开放：

- 自动检查更新。
- 手动重新检查。
- 打开 GitHub Release 页面。
- 展示当前桌面版本、当前镜像 tag、最新版本、更新时间。

`v0.5.0` 可预留但不默认开放：

- 一键更新镜像栈。
- 下载 zip 并校验 sha256。

暂不实现：

- 自动替换 portable zip。
- 静默更新。
- 无确认数据库迁移。
- 自动回滚数据库。

## 5. Release Manifest

### 5.1 生成位置

Release workflow 在 GitHub Release 上传 portable zip 时同步上传 manifest。

推荐资产名：

- `yunwu-release.json`

桌面壳获取 latest 版本时优先使用 GitHub Releases API：

```text
GET https://api.github.com/repos/Alterego-42/Yunwu-custom-api/releases/latest
```

然后在 latest release assets 中查找 `yunwu-release.json`。这样不需要额外托管 `latest.json`。

### 5.2 Schema

`v0.5.0` 推荐 manifest：

```json
{
  "schemaVersion": 1,
  "appId": "ai.yunwu.desktop",
  "channel": "stable",
  "version": "0.5.0",
  "tag": "v0.5.0",
  "commit": "git-sha",
  "publishedAt": "2026-06-06T00:00:00Z",
  "releaseUrl": "https://github.com/Alterego-42/Yunwu-custom-api/releases/tag/v0.5.0",
  "releaseNotes": {
    "summary": "Adds batch concurrent image requests and update-aware desktop baseline.",
    "url": "https://github.com/Alterego-42/Yunwu-custom-api/releases/tag/v0.5.0"
  },
  "desktop": {
    "version": "0.5.0",
    "minSupportedVersion": "0.5.0",
    "requiresDesktopUpdate": true,
    "portableAsset": {
      "name": "Yunwu.Desktop-0.5.0-win-x64-portable.zip",
      "sha256": "sha256-hex",
      "size": 0
    }
  },
  "stack": {
    "imageOnlySupported": false,
    "composeSchemaVersion": 2,
    "runtimeEnvVersion": 2,
    "images": {
      "api": "ghcr.io/alterego-42/yunwu-custom-api-api:v0.5.0",
      "worker": "ghcr.io/alterego-42/yunwu-custom-api-worker:v0.5.0",
      "web": "ghcr.io/alterego-42/yunwu-custom-api-web:v0.5.0"
    }
  },
  "migration": {
    "hasDatabaseMigration": true,
    "risk": "additive",

      "Adds task_batch_items for batch execution slots.",

  }
}
```

后续只更新镜像的 `v0.5.x` manifest 可以设置：

```json
{
  "desktop": {
    "requiresDesktopUpdate": false,
    "minSupportedVersion": "0.5.0"
  },
  "stack": {
    "imageOnlySupported": true,
    "composeSchemaVersion": 2,
    "runtimeEnvVersion": 2
  }
}
```

### 5.3 校验规则

桌面壳必须校验：

- `schemaVersion === 1`
- `appId === "ai.yunwu.desktop"`
- `channel === "stable"`，除非用户显式配置其他 channel。
- `tag` 符合 `vX.Y.Z`。
- `releaseUrl` host 为 `github.com`，repo 为 `Alterego-42/Yunwu-custom-api`。
- `portableAsset.name` 以 `Yunwu.Desktop-` 开头并以 `-win-x64-portable.zip` 结尾。

不要执行 manifest 中的任何命令，不要打开任意 URL。

## 6. Desktop Runtime State

新增持久化文件：

```text
app.getPath("userData")/runtime/release-state.json
```

建议结构：

```json
{
  "schemaVersion": 1,
  "desktopVersion": "0.5.0",
  "bundledImageTag": "v0.5.0",
  "currentImageTag": "v0.5.0",
  "currentManifestTag": "v0.5.0",
  "lastCheckedAt": "2026-06-06T00:00:00Z",
  "lastAppliedAt": null,
  "lastKnownLatest": null,
  "lastError": null
}
```

读取规则：

- 无文件时，以 `app.getVersion()` 推导 `bundledImageTag=v${app.getVersion()}`。
- 如果用户手动安装了更高版本桌面包，且 `currentImageTag` 低于 `bundledImageTag`，默认前进到 `bundledImageTag`。
- 如果未来支持用户 pin 旧镜像，再新增 `pinnedImageTag`；`v0.5.0` 不支持 pin。
- 生成 runtime `.env` 时读取 `currentImageTag`，不要继续硬编码版本。

写入 runtime `.env`：

```env
YUNWU_IMAGE_REGISTRY=ghcr.io/alterego-42
YUNWU_IMAGE_TAG=<currentImageTag>
TASK_WORKER_CONCURRENCY=50
TASK_BATCH_WORKER_CONCURRENCY=2
TASK_BATCH_QUEUE_NAME=yunwu-image-batch-tasks
```

## 7. Desktop Update Service

建议在 `apps/desktop/src/main` 下拆分小模块，避免继续扩大 `main.ts`：

- `release-state.ts`：读写 `release-state.json`。
- `release-manifest.ts`：拉取、解析、校验 manifest。
- `update-service.ts`：比较版本、产出状态、执行允许的更新动作。
- `compose-runtime.ts`：后续可从 `main.ts` 抽出 Compose 文件路径和运行命令。

如果 `v0.5.0` 为了控制改动范围暂不拆文件，也应保持这些职责的函数边界。

### 7.1 状态类型

更新状态建议：

```ts
type UpdatePhase =
  | "unknown"
  | "checking"
  | "up-to-date"
  | "desktop-update-required"
  | "image-update-available"
  | "blocked"
  | "applying"
  | "applied"
  | "error";
```

Renderer 可消费的状态：

```ts
type UpdateStatus = {
  phase: UpdatePhase;
  currentDesktopVersion: string;
  currentImageTag: string;
  latestVersion?: string;
  latestTag?: string;
  releaseUrl?: string;
  canOpenReleasePage: boolean;
  canApplyImageUpdate: boolean;
  requiresDesktopUpdate: boolean;
  message: string;
  checkedAt?: string;
  error?: string;
};
```

### 7.2 版本比较

规则：

- `v0.5.0` 和 `0.5.0` 统一归一化为 semver。
- latest <= current desktop version 且 latest <= current image tag 时：`up-to-date`。
- latest > current desktop version 且 manifest `requiresDesktopUpdate=true`：`desktop-update-required`。
- manifest `imageOnlySupported=true` 且 current desktop >= `minSupportedVersion`：`image-update-available`。
- manifest `updateBlocked=true` 或 schema 不兼容：`blocked`。

### 7.3 IPC

Preload 只暴露受限能力：

```ts
contextBridge.exposeInMainWorld("yunwuDesktop", {
  getStatus: ...,
  getUpdateStatus: () => ipcRenderer.invoke("desktop:get-update-status"),
  checkUpdates: () => ipcRenderer.invoke("desktop:check-updates"),
  openReleasePage: () => ipcRenderer.invoke("desktop:open-release-page"),
  applyImageUpdate: () => ipcRenderer.invoke("desktop:apply-image-update")
});
```

安全约束：

- `openReleasePage` 只能打开已校验 manifest 的 release URL。
- `applyImageUpdate` 不接收任意 tag 或 URL，由主进程使用已校验 manifest。
- 不暴露任意 shell、Docker、文件路径或 URL 参数。
- Web 页面只展示状态和触发固定动作。

## 8. Image Stack Update Flow

`v0.5.0` 可以先实现状态机和禁用按钮；后续打开时按此流程执行。

前置条件：

- `canApplyImageUpdate=true`
- Docker CLI 和 daemon 可用。
- 当前没有 `startStack`、`stopCompose` 或其他 update 流程运行。
- 用户确认“会重启本地服务，数据卷保留，数据库迁移可能前进”。

流程：

1. 记录 previous image tag。
2. 将 candidate tag 写入 runtime `.env`，但 release state 标记为 pending。
3. 执行 `docker compose pull api worker web`。
4. pull 失败：恢复 `.env` previous tag，状态为 `error`，不改 `currentImageTag`。
5. pull 成功后执行当前实例 `docker compose down`，保留 volume。
6. 将 `currentImageTag` 提交为 candidate tag，并重写 runtime `.env`。
7. 执行 `docker compose up -d`。
8. 等待 API `/health`、API `/readiness`、Web `/health`。
9. 成功：状态为 `applied`，记录 `lastAppliedAt`。
10. 失败：状态为 `error`，保留 candidate tag，不自动回滚，提示查看日志和 release notes。

不自动回滚的原因：

- API 启动时可能已经执行 `prisma migrate deploy`。
- 数据库迁移通常只能保证前向兼容，不能保证降级。
- 自动切回旧镜像可能让旧代码面对新 schema，风险更高。

## 9. `v0.5.0` Release Checklist

版本同步：

- 根 `package.json`: `0.5.0`
- `apps/api/package.json`: `0.5.0`
- `apps/web/package.json`: `0.5.0`
- `apps/desktop/package.json`: `0.5.0`
- `packages/shared/package.json`: `0.5.0`
- `.github/workflows/release.yml` manual dispatch 默认值：`v0.5.0`
- `apps/desktop/src/renderer/index.html` 不再硬编码 `v0.4.3`，改为主进程状态注入或 `app.getVersion()` 显示。
- `apps/desktop/src/main/main.ts` 不再硬编码 `YUNWU_IMAGE_TAG=v0.4.3`。
- `infra/docker-compose.desktop.yml` 默认 tag 更新为 `v0.5.0`，或改为由 runtime `.env` 强制提供。

Release workflow：

- Docker job 继续构建 `api`、`worker`、`web` 三个 target。
- Windows job 继续生成 portable zip。
- Release job 下载 portable artifact。
- Release job 计算 portable zip sha256 和 size。
- Release job 生成 `yunwu-release.json`。
- Release job 上传 portable zip 和 `yunwu-release.json`。

Release notes 必须明确：

- `v0.5.0` 是批量并发任务版本。
- 从 `v0.4.3` 升级需要下载新的 portable zip。
- 数据卷保留；首次启动会执行数据库迁移。
- 登录态应保留，因为 `userData/runtime/identity.json` 保留。
- 批量任务会提高普通任务并发，用户应注意上游 provider 限流。

## 10. 测试计划

自动化：

```powershell
pnpm --filter @yunwu/desktop typecheck
pnpm desktop:verify-release
pnpm --filter @yunwu/api exec node --require ts-node/register --test "test/*.test.ts" "src/logging/app-logger.service.spec.ts"
pnpm --filter @yunwu/web test:routes
```

新增更新策略测试建议：

- manifest parser 接受合法 `v0.5.0` manifest。
- manifest parser 拒绝错误 `appId`、错误 host、非 semver tag。
- version comparator 正确判断 `desktop-update-required`。
- release state 无文件时默认使用 `v${app.getVersion()}`。
- release state 在桌面包版本前进时同步到新 bundled tag。
- `writeRuntimeFiles` 使用 release state 中的 `currentImageTag`。

手动升级验收：

1. 用 `v0.4.3` 桌面包创建用户、生成作品、关闭桌面。
2. 启动 `v0.5.0` portable 包。
3. 确认 Docker volume 未删除，旧用户仍可登录。
4. 确认 API migration 成功，`task_batch_items` 存在。
5. 提交 `batchCount=3` 任务，确认一个批量任务卡和三个槽。
6. 关闭桌面，确认当前 Compose 项目 down，volume 保留。
7. 再次启动，确认仍使用 `v0.5.0` image tag。

后续镜像栈更新验收：

1. 使用测试 manifest 指向 `v0.5.1`。
2. 检查状态为 `image-update-available`。
3. 确认用户确认后执行 pull/down/up。
4. 健康检查通过后状态为 `applied`。
5. pull 失败时 `.env` 恢复旧 tag。
6. up 失败时不自动回滚，并提示日志位置。

## 11. 分阶段交付

### Phase A：`v0.5.0` 必做

- 统一版本到 `0.5.0`。
- 桌面 runtime image tag 从 `app.getVersion()` 或 release state 推导。
- 新增 release state 文件。
- 新增 release manifest 生成和上传。
- 桌面启动页显示当前桌面版本和当前镜像 tag。
- 桌面启动页支持检查更新和打开 release 页面。
- `v0.5.0` manifest 标记 `requiresDesktopUpdate=true`。

### Phase B：`v0.5.x` 打开

- 完成 `image-update-available` 状态。
- 完成镜像栈更新确认和执行。
- Web 设置页或管理页展示更新状态。
- 支持测试 manifest URL 环境变量。

### Phase C：installer 评估

- 如果用户需要真正一键替换桌面程序，再新增 NSIS target。
- 保留 portable zip 给高级用户。
- 使用 `electron-updater` 生成 `latest.yml`。
- 迁移前仍执行当前 Compose 项目 down，保留 volume。

## 12. 决策摘要

- `v0.5.0` 对 `v0.4.3` 用户是手动桌面包升级。
- `v0.5.0` 是后续用户侧更新能力的起点。
- 更新逻辑属于 Electron 桌面壳，不属于 API 账号系统。
- 镜像栈更新可以做，但必须由 manifest 明确兼容，且必须有用户确认。
- 数据库迁移只承诺前向升级，不承诺自动回滚。
- Portable zip 自替换不作为当前优先路径；需要完整自动更新时再引入 installer。
